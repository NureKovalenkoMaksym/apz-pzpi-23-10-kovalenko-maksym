3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
    Шаблони (патерни) проєктування — це перевірені часом типові рішення поширених проблем у проєктуванні архітектури програмного забезпечення. На відміну від готових функцій чи бібліотек, шаблон не є конкретним фрагментом коду, а радше загальною концепцією вирішення проблеми. Каталог класичних патернів проєктування був формалізований у книзі «Design Patterns: Elements of Reusable Object-Oriented Software» (автори відомі як GoF — «Банда чотирьох»). Відповідно до класифікації GoF, шаблони поділяються на три основні категорії:
• Породжувальні (Creational): забезпечують механізми створення об'єктів, що підвищують гнучкість системи та повторне використання коду.
• Структурні (Structural): визначають способи збирання об'єктів та класів у більші структури, забезпечуючи їхню ефективність.
• Поведінкові (Behavioral): визначають алгоритми та способи ефективної комунікації між об'єктами. 
3.2 Опис обраного патерна  
Назва патерна: Memento (Зберігач).
Тип шаблона (патерна): Поведінковий.
     Основне призначення: Патерн Memento забезпечує механізм збереження та відновлення попереднього стану об'єкта таким чином, щоб не порушувати принцип інкапсуляції. Він дозволяє створювати «знімки» поточного стану об'єкта і зберігати їх у зовнішньому середовищі без розкриття деталей його внутрішньої реалізації.
     Передумови застосування: Використання шаблона є доцільним у випадках, коли:
1. Необхідно зафіксувати «знімок» стану об'єкта або його частини для подальшого відновлення (наприклад, реалізація механізму Undo/Redo, збереження прогресу).
2. Пряме отримання стану об'єкта через публічні інтерфейси розкриває його внутрішню структуру та порушує правила інкапсуляції.
Переваги:
• Не порушує принцип інкапсуляції, оскільки зовнішні об'єкти не мають доступу до приватного стану ініціатора.
• Спрощує структуру класу-ініціатора, переносячи відповідальність за збереження історії станів на клас-опікун.
Недоліки:
• Може потребувати значних обсягів оперативної пам'яті, якщо стан об'єкта великий, а збереження відбуваються занадто часто.
• Вимагає додаткових обчислювальних ресурсів на створення копій об'єктів.
 
3.3  Структура шаблона (патерна) 

Структура патерна Memento базується на взаємодії трьох основних компонентів:
• Originator (Ініціатор): об'єкт, поточний стан якого потрібно зберегти. Ініціатор містить метод для створення об'єкта-зберігача (Memento), а також метод для відновлення свого стану з отриманого зберігача.
• Memento (Зберігач): об'єкт-капсула, що містить «знімок» внутрішнього стану ініціатора. Зберігач має закритий інтерфейс для всіх об'єктів, окрім свого ініціатора.
• Caretaker (Опікун): об'єкт, що відповідає за зберігання та надання доступу до об'єктів-зберігачів. Опікун ніколи не змінює і не переглядає внутрішній вміст зберігачів.

    
     Рисунок 3.1 – UML-діаграма структури патерна Memento)
 	Коли виникає потреба зберегти стан, Опікун звертається до Ініціатора з вимогою створити Зберігача. Ініціатор генерує новий об'єкт Зберігача, заповнюючи його своїми поточними параметрами, та передає його Опікуну. Для відновлення стану Опікун передає відповідний об'єкт Зберігача назад Ініціатору, який розпаковує дані.
 
3.4 Реалізація шаблона (патерна)
Опис реалізації мовою програмування: C#. Програмна реалізація патерна Memento поділена на три логічні блоки:
1. Створення базових структур: Розроблено клас GameSave (Зберігач). Він містить властивості для збереження параметрів здоров'я (Health) та розташування (Location) об'єкта. Властивості доступні лише для читання зовнішніми об'єктами (private set).
2. Реалізація конкретних класів: Створено клас Player (Ініціатор), який імітує сутність у програмній системі. Клас містить метод SaveState(), що упаковує поточний стан у новий об'єкт GameSave, та RestoreState(), що приймає об'єкт для відновлення полів класу.
3. Демонстрація використання: Розроблено клас SaveManager (Опікун), який містить колекцію типу List<GameSave> для зберігання історії. У головному методі програми (Main) продемонстровано ініціалізацію стану, збереження його за допомогою Опікуна, зміна стану та його успішне відновлення.

42         _history.Add(save);
43     }
44 
45     public GameSave GetSave(int index)
46     {
47         return _history[index];
48     }
49 }
